That points to a very specific answer:

Recommendation
The simplest bootstrapping approach is not to start with “Micro Emacs” directly.
It is to build:

a tiny buffer core in C or assembler

a command loop that is essentially an ed-like line editor with Emacs-flavored names

a minimal display layer that renders exactly 3 lines:

current edit line / view line

mode line

minibuffer / command input

an internal command dispatch structure designed so later:

today: commands are native C/asm functions

later: commands can be rebound to Lisp symbols/functions

So the real bootstrap is:

Stage 0: native core + native commands
Stage 1: native core + command table exposed to Lisp
Stage 2: editing logic increasingly moved into Lisp
Stage 3: Lisp-defined commands/macros/keymaps, native primitives underneath

This is the least risky path because the editor core needs a few things that are painful to bootstrap entirely in Lisp too early:

memory management

buffer mutation

cursor motion

span deletion/insertion

display rendering

command parsing from UART

High-level architecture
You only need four subsystems for MVP:

1. Buffer model
Represents editable text and cursor position.

2. Command interpreter
Reads a line from UART, parses it, executes one command.

3. Three-line renderer
After every command, redraw:

text line

status line

minibuffer/result line

4. Persistence/copy-back
For now:

source buffer at fixed address

working copy in heap-managed editable storage

save copies working buffer back to original address

That is enough.

Nonfunctional goals for the bootstrap
You want the design to survive the transition to Lisp, so optimize for:

Simplicity

Inspectability
You should be able to dump state over UART and debug it easily.

Replaceability
Anything that will later become Lisp should sit behind a tiny command interface.

Bounded memory behavior
Because this is an embedded target, make failures explicit:

Simplest viable internal design
For bootstrapping, I strongly recommend:

Use a gap buffer
Not a rope, not a piece table, not a per-line linked list.

A gap buffer is the best fit here because:

Benefits
simple to implement in C or assembler

efficient local insert/delete around cursor

natural cursor model

later still usable under Lisp primitives

easy to save by linearizing around the gap

Costs
moving cursor long distances requires moving bytes

line count / column can be expensive unless cached

Those costs are fine for MVP.

Yes — a repeat count / universal-argument-like prefix belongs in the MVP.

It is not just a convenience feature; for a UART-friendly editor it is one of the main ways to keep the UI usable when you do not yet have rich screen control or full keymaps.

Why it belongs in MVP
Your interface has three constraints:

serial/UART friendly

very small command set

repaint only in a minimal 3-line UI

That means users will often need to:

move many characters or lines

insert repeated characters

delete multiple chars/words/lines

open vertical space with repeated newlines

Without a repeat mechanism, even basic tasks become tedious.

So I agree: put repeat in from the start.

Recommendation
Use a pending repeat count in editor state.

Conceptually:

default repeat = 1

Ctrl-U enters “repeat prefix collection”

digits accumulate the count

the next repeatable command consumes it

then repeat resets back to 1

So examples become:

Ctrl-U 80 right

Ctrl-U 80 Enter

Ctrl-U 10 backspace

Ctrl-U 5 down

That is the right model.

You wrote:

That feels natural in prose, but the cleaner implementation is the Emacs-like order:

Best semantics for your editor
Because you have both edit mode and command mode, repeat should work in both.

In edit mode
Repeat applies to the next editing keystroke or motion keystroke.

Examples:

Ctrl-U 80 * inserts 80 asterisks

Ctrl-U 80 Enter inserts 80 newlines

Ctrl-U 12 right moves 12 chars right

Ctrl-U 5 backspace deletes 5 chars backward

In command mode
Repeat applies to the next command if that command is repeatable.

Examples:

repeat 80 right

or internally, count=80 before right

goto-line probably should not be repeated

save should not be repeated

This gives one unified internal mechanism.

Internal design
Add these fields to editor state:

struct RepeatState {

Thought for 20s
Level-setting: Ctrl-U is not a UART feature. It is just the single control byte 0x15 (“NAK” in ASCII). Whether that byte reaches your COR24 program depends on every layer in between: OS tty line discipline, terminal emulator, telnet mode, browser UI, and platform/global shortcuts. Telnet itself is fundamentally an 8-bit byte-oriented channel, so it can carry 0x15 just fine; the problem is usually that something local intercepts it first. 

What usually goes wrong
On Unix-like hosts, terminals are often in canonical/cooked mode unless the application switches them out of it. In canonical mode, special control characters such as VKILL are interpreted by the tty layer instead of being passed through. On many systems, kill = ^U, which is exactly why Ctrl-U often erases the current input line instead of reaching the target app. termios/stty expose these special-character settings, including VKILL, VERASE, VINTR, and friends. 

That is also why you “cannot type Ctrl-U here”: this chat input, like many web text inputs, is not a raw terminal and may handle control combinations locally instead of sending them as characters.

What a telnet client can send
A telnet connection can carry control bytes, including 0x15, because Telnet is defined as a bidirectional byte stream. But telnet clients may operate in either:

character-at-a-time mode, where keys are sent immediately, or

linemode, where the client does local line editing and only sends completed lines or edited results.

Telnet linemode explicitly allows client-side editing, which is exactly the kind of thing that can swallow Ctrl-U before the remote end sees it. 

So: telnet is not a magic fix. It can work, but only if you are in a mode where local editing is disabled or limited enough that 0x15 actually gets through.

Historical way around this
Yes — there is a historical pattern for avoiding fragile control-key delivery:

Use an escape/prefix character in-band
Examples from old systems and editors:

a command introducer like : or ?

control-byte quoting, where one prefix means “the next thing is literal” or “the next thing is a command”

This is historically robust because printable ASCII tends to survive more layers than raw control chords.

# sw-yocto-ed — Product Requirements Document (PRD)

## Overview

sw-yocto-ed is a minimal modal text editor designed for the COR24 ISA environment. It operates over UART using a transport-safe ASCII command interface and a simple 3-line display model.

The editor is intended as a bootstrap system that will evolve toward a programmable, Lisp-driven editor.

---

## Goals

### Primary Goals
- Provide a usable text editor over UART with minimal dependencies
- Support basic text navigation, insertion, deletion, and saving
- Use a transport-safe command interface (ASCII only)
- Establish a foundation for future Macro Lisp integration

### Secondary Goals
- Enable scripting and automation via textual commands
- Provide a clean path to host-side integrations (Rust client, Emacs mode)

---

## Non-Goals (MVP)
- Full Emacs compatibility
- Multiple buffers
- Regions/mark system
- Undo/redo
- Syntax highlighting
- Full-screen terminal control (VT100, curses)

---

## User Interaction Model

### Modes

#### Edit Mode
- Default mode
- Printable characters insert at point
- Enter inserts newline
- Backspace deletes previous character
- Escape switches to Command Mode

#### Command Mode
- Accepts ASCII commands on input line
- Enter executes command
- Returns to Edit Mode after execution
- `C-]` cancels and returns to Edit Mode

---

## Display Model (3-line UI)
TEXT> current line with | indicating point
MODE> INS row:X col:Y lines:N size:S dirty|clean
CMD > command input or status message


---

## Core Features (MVP)

### Navigation
- left, right
- up, down
- beginning/end of line (bol/eol)
- beginning/end of buffer (bob/eob)
- forward/backward word (fwdw/backw)

### Editing
- insert (edit mode typing)
- insert TEXT (command mode)
- newline
- delete-next-char (del)
- delete-prev-char (backdel)
- delete-line (delline)

### Buffer
- save
- new (clear buffer)
- quit

### Utility
- goto N (line number)
- help

---

## Repeat / Count Prefix

Textual prefix:
8 right
80 insert "*"
10 newline


Semantics:
- Prefix applies to command execution count
- Applies to motion, deletion, and insert commands
- Default count = 1

Future:
- optional Ctrl-U prefix for interactive use

---

## Data Model

- Single buffer
- Gap buffer representation
- Fixed memory regions
- Dirty flag
- Cursor (point)

---

## Success Criteria

- User can edit, navigate, and save text via UART
- Commands are consistent and predictable
- System runs within COR24 constraints
- Codebase supports future extension (Lisp, host UI)

---

## Future Directions

- Macro Lisp command binding
- Host-side Rust terminal bridge
- Emacs remote editing mode
- Regions and kill/yank
- Multiple buffers

## Overview

The editor is structured as a small, layered system designed for simplicity, extensibility, and low resource usage.

+----------+-----------+

| (Edit / Command) |
+----------+-----------+
|
+----------v-----------+

+----------+-----------+

+----------+-----------+
|
+----------v-----------+

+----------------------+

## Components

### 1. UART Interface
- Reads input characters
- Writes output lines
- No terminal control assumptions

---

### 2. Input State Machine

States:
- EDIT
- COMMAND

Responsibilities:
- Route input to appropriate handler
- Manage transitions (Escape, Enter, C-])

---

### 3. Command Dispatcher

Input:
[count] command [args...]


Output:
- Executes command N times
- Returns status/result

---

### 4. Editor Core

Maintains:
- Buffer (gap buffer)
- Cursor (point)
- Dirty flag

Provides primitives:
- insert_char
- insert_string
- delete_forward
- delete_backward
- move_left/right/up/down
- line/word scanning

---

### 5. Renderer

Repaints all 3 lines after each command:
- TEXT line
- MODE line
- CMD line

No partial redraw optimization in MVP

---

## Data Structures

### Buffer
[ text_before ][ GAP ][ text_after ]
^
point


Fields:
- base pointer
- capacity
- gap_start
- gap_end
- dirty flag

---

## Command Execution Flow
Input → Parse → CommandInvocation
→ Apply repeat count
→ Execute command(s)
→ Update state
→ Render


---

## Error Handling

- Commands return status codes
- Renderer displays error messages in CMD line
- Partial operations allowed for movement
- Insert operations may pre-check capacity

---

## Extensibility

Future layers:
- Lisp command bindings
- Host-side protocol adapters
- richer UI

Architecture intentionally keeps:
- core editing primitives native
- higher-level behavior pluggable

# sw-yocto-ed — Detailed Design

## Input Handling

### Edit Mode

| Input        | Action                  |
|-------------|--------------------------|
| printable   | insert char              |

| Escape      | switch to command mode   |

---

### Command Mode

| Input        | Action                      |
|-------------|-----------------------------|
| text        | build command string        |
| Backspace   | edit command                |
| Enter       | execute command             |
| C-]         | cancel, return to edit mode |

---

## Command Grammar
[count] command [arguments...]


Examples:
right
8 right
insert foo
80 insert "*"
goto 42


---

## Command Table

| Command   | Description |
|----------|------------|
| left/right/up/down | movement |
| bol/eol | line boundaries |
| bob/eob | buffer boundaries |
| fwdw/backw | word movement |
| del/backdel | char delete |
| delline | delete line |
| newline | insert newline |
| insert TEXT | insert string |
| save | save buffer |
| new | clear buffer |
| quit | exit |
| goto N | jump to line |
| help | show help |

---

## Repeat Semantics

- Default count = 1
- Prefix modifies command execution count
- Applies to repeatable commands only

Execution model:
for i in 1..count:
command()


---

## Buffer Operations

### Insert Char
- move gap if needed
- place char at gap_start
- increment gap_start

### Delete Forward
- expand gap_end

### Delete Backward
- decrement gap_start

### Move Cursor
- shift gap left/right

---

## Line Handling

- Lines separated by '\n'
- BOL/EOL found via scan
- Row/column computed on demand

---

## Mode Line Fields
MODE> [INS|CMD] row:X col:Y lines:N size:S dirty|clean


---

## Rendering

Always redraw:

1. TEXT line
   - current line only
   - insert `|` at point

2. MODE line

3. CMD line

---

## Memory Layout (conceptual)
0x000000 Editor code
0x010000 Source buffer
0x020000 Heap (gap buffer)
0x040000 Stack

## Future Hooks

- mark position (reserved, unused in MVP)
- kill buffer
- command registry for Lisp
